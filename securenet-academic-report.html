Section 01

Executive Summary

This report documents the complete design and simulated implementation of an enterprise network for SecureNet Technologies. Acting in the role of Junior Network Security Engineer, this project delivers a robust, secure, and scalable network infrastructure across one headquarters site and two remote branch locations.

The design follows Cisco's proven three-tier hierarchical model comprising Core, Distribution, and Access layers. The network incorporates eight segmented VLANs, redundant distribution-layer switching with Hot Standby Router Protocol (HSRP) and IEEE 802.3ad EtherChannel, three distinct routing protocols (OSPF at HQ core, RIPv2 at branches), site-to-site IPsec VPN tunnels secured by IKEv1, a demilitarised zone (DMZ) protected by a Cisco ASA 5506-X firewall, centralised authentication via RADIUS/AAA, and wireless isolation for guest users. All components have been simulated and validated in Cisco Packet Tracer 9.0.

Key Outcome: A 45-device topology with zero single points of failure at the distribution layer, full inter-VLAN routing, dual IPsec VPN tunnels to branch sites, and a hardened DMZ hosting Web, Email, and FTP public services.

  • Provide high availability at the distribution layer so that no single switch failure disrupts connectivity
  • Connect two geographically separate branch offices to headquarters via encrypted tunnels over public infrastructure
  • Expose selected public services (Web, Email, FTP) to the internet without compromising the internal network
  • Provide a guest wireless network isolated from corporate resources

  • Maintain comprehensive monitoring via SNMP, Syslog, NTP, and DNS

3.1 Core Layer

A single Cisco 2911 router (HQ-Core) serves as the core routing device. It connects directly to the ASA firewall on its inside interface and maintains Layer 3 routed links to both distribution switches. Running OSPF Area 0 with default-information originate, it propagates a default route learned from the ASA throughout the HQ network. The core router also integrates RADIUS-based AAA for device management access.

3.2 Distribution Layer

Two Cisco 3560-24PS Layer 3 switches (HQ-Dist1 and HQ-Dist2) form the distribution layer. These perform inter-VLAN routing via Switched Virtual Interfaces (SVIs), run HSRP for gateway redundancy, form an EtherChannel peer-link for VLAN traffic sharing, and enforce Spanning Tree root priorities aligned with HSRP active roles. Both switches participate in OSPF Area 0 to advertise VLAN subnets to the core.

3.3 Access Layer

Six Cisco 2960-24TT Layer 2 switches provide end-user connectivity, one per department VLAN. Each access switch connects to both distribution switches via dual trunked uplinks (Fa0/23 → Dist1, Fa0/24 → Dist2). Spanning Tree Protocol blocks the redundant path under normal operation and automatically restores it on upstream failure. All host-facing ports are configured with spanning-tree portfast and bpduguard enable to eliminate convergence delay and prevent rogue switch attachment.

Section 05

Routing Protocol Implementation

Three routing protocols are deployed across the network, each selected for the specific characteristics of the network segment it serves. This multi-protocol approach demonstrates the strengths and appropriate use cases of each protocol.

5.1 OSPF — Open Shortest Path First (HQ Core)

OSPF is deployed as a single Area 0 across the HQ core and distribution layers. As a link-state protocol, OSPF maintains a complete topology map (LSDB) and uses Dijkstra's algorithm to compute the shortest path to each destination. This makes OSPF fast-converging and loop-free by design.

Table 2 — OSPF Configuration Summary
DeviceRouter-IDNetworks Advertised
HQ-Core1.1.1.110.0.0.0/30, 10.10.0.0/30, 10.10.0.4/30
HQ-Dist12.2.2.210.10.0.0/30, 10.10.10–60/24, 10.0.1.0/24
HQ-Dist23.3.3.310.10.0.4/30, 10.10.40–60/24

HQ-Core uses default-information originate to propagate its default route (pointing to the ASA) to all OSPF neighbours, ensuring all internal subnets have a path to the internet without requiring static routes on distribution switches.

Design decision: Only the HSRP Active switch for a given VLAN advertises that SVI subnet into OSPF. Both advertising the same subnet would create duplicate paths and asymmetric routing. This is enforced by including the subnet in OSPF only on the Active device.

5.2 RIPv2 — Routing Information Protocol (Branches & ISP)

RIPv2 is deployed at the branch sites and on the ISP router. As a distance-vector protocol with a maximum hop count of 15, RIPv2 is simple to configure and appropriate for small branch environments with limited topology complexity. no auto-summary is configured to prevent automatic summarisation to classful boundaries, which would cause routing black holes with the discontiguous subnets in use.

5.3 BGP — Border Gateway Protocol (Discussion)

While BGP is not implemented in the simulated topology (Cisco Packet Tracer does not support iBGP and real BGP peering with an ISP requires autonomous system assignment), it is included in this report as the real-world routing protocol that would be deployed between SecureNet's ASA edge and the ISP.

BGP is the protocol of the internet, operating between autonomous systems (AS) to exchange reachability information. In a production deployment, SecureNet would be assigned a public AS number and would establish eBGP peering with the ISP to advertise its public address block. BGP's policy controls (route maps, communities, local preference, MED) would allow SecureNet to implement traffic engineering and multi-homing.

Section 06

Redundancy — HSRP, EtherChannel & STP

Network redundancy eliminates single points of failure. The distribution layer implements three complementary redundancy mechanisms that operate at different OSI layers.

6.1 HSRP — Hot Standby Router Protocol (Layer 3)

HSRP is a Cisco proprietary First Hop Redundancy Protocol (FHRP). It allows two physical switches to present a single virtual IP address and MAC address to end hosts. End hosts configure this virtual IP as their default gateway. If the Active switch fails, the Standby switch takes over the virtual IP within seconds without requiring any host reconfiguration.

The VLAN-to-active-switch assignment is load-balanced across both distribution switches:

Table 3 — HSRP Load Balancing
SwitchHSRP Active (priority 110)HSRP Standby (priority 90)
HQ-Dist1VLAN 10, 20, 30, 80VLAN 40, 50, 60
HQ-Dist2VLAN 40, 50, 60, 70VLAN 10, 20, 30, 80

The preempt keyword is configured on all Active groups, ensuring that if the Active switch recovers after a failure it immediately reclaims the Active role rather than remaining in Standby.

6.2 EtherChannel — IEEE 802.3ad LACP (Layer 2)

EtherChannel bundles multiple physical links into a single logical interface, providing both redundancy and bandwidth aggregation. Port-Channel 1 is formed between HQ-Dist1 and HQ-Dist2 using LACP (Link Aggregation Control Protocol) with two FastEthernet links (Fa0/23 and Fa0/24), creating a logical 200 Mbps peer-link.

This peer-link is critical for HSRP operation: it carries HSRP hello messages between Dist1 and Dist2. Without it, both switches would simultaneously believe they are Active (split-brain), causing a duplicate gateway MAC on the network. The peer-link trunk is configured to allow all VLANs (10–80).

6.3 Spanning Tree — Rapid PVST+ (Layer 2)

All switches run Rapid Per-VLAN Spanning Tree (Rapid-PVST+). STP root priorities are deliberately aligned with HSRP Active assignments to prevent suboptimal forwarding paths. If STP root and HSRP Active were on different switches, traffic would cross the peer-link unnecessarily, increasing latency and consuming bandwidth.

Table 4 — STP Root Priority Alignment with HSRP
SwitchSTP Root VLANs (priority 4096)STP Non-Root VLANs (priority 8192)
HQ-Dist110, 20, 30, 8040, 50, 60, 70
HQ-Dist240, 50, 60, 7010, 20, 30, 80

Access switch host-facing ports use spanning-tree portfast to bypass the 30-second STP listening/learning delay and bpduguard enable to automatically err-disable the port if a BPDU is received — preventing unauthorised switch connections from disrupting the spanning tree topology.

Section 07

Security Design — Firewall, DMZ & ACLs

The security architecture places a Cisco ASA 5506-X firewall as the single choke point between the untrusted internet, the internal network, and the semi-trusted DMZ. This creates three distinct security zones with enforced traffic policies between them.

7.1 DMZ Architecture

The Demilitarised Zone hosts three public-facing servers: a web server (HTTP/HTTPS), a mail server (SMTP/POP3), and an FTP server. These servers must be reachable from the internet but must not have direct access to internal corporate resources.

Each DMZ server is mapped to a public IP via static NAT on the ASA. This allows external users to reach services using public addresses while the servers retain private RFC 1918 addresses internally.

7.3 Access Control Lists

Two ACLs enforce traffic policy on the ASA. The outside-in ACL permits only specific service ports inbound from the internet to the static NAT public IPs. All other inbound traffic is implicitly denied. The dmz-in ACL restricts DMZ-initiated traffic to the inside network, permitting only reply traffic using source-port matching (required due to Packet Tracer's ASA not supporting the established keyword).

PT Limitation: The ASA in Packet Tracer silently drops ACE entries using the established keyword. The dmz-in ACL uses explicit source-port matching (e.g. permit tcp 172.16.0.0/24 eq 80 10.0.0.0/8) as the workaround. This is functionally equivalent for the simulated environment.

7.4 PAT for Internal Users

All internal hosts (10.0.0.0/8) share the ASA's outside interface IP for internet access via dynamic NAT overload (Port Address Translation). This conserves public IP addresses and hides internal addressing from the internet.

Section 09

Network Services — DHCP, DNS, NTP, AAA

9.1 DHCP

Dynamic Host Configuration Protocol is implemented in two locations. The distribution layer switches serve as DHCP relay agents using ip helper-address 10.0.1.10 on every user-facing SVI. This forwards broadcast DHCP requests from clients to the centralised DHCP server (SRV-Internal) across the routed infrastructure. A separate DHCP pool for Guest WiFi clients is hosted locally on HQ-Dist2's VLAN 70 SVI.

9.2 DNS — Domain Name System

A dedicated DNS-Server (10.0.1.11) in VLAN 80 provides name resolution for all internal and external hostnames. All devices reference this server. DNS records map internal hostnames to private IPs and external hostnames to the public NAT addresses of DMZ servers. All PCs and servers are configured with DNS pointing to 10.0.1.11.

9.3 NTP — Network Time Protocol

SRV-Internal acts as the NTP master for the network. All routing devices (ASA, Core, Dist1, Dist2, Branch routers) reference it with ntp server 10.0.1.10. Synchronised timestamps are critical for log correlation during security incident investigation and for certificate validity checking in IPsec operations.

9.4 AAA — Authentication, Authorisation & Accounting

A dedicated AAA-Server (10.0.1.12) runs RADIUS on ports 1645 (authentication) and 1646 (accounting). Network devices authenticate management access against this server rather than using local usernames. HQ-Core uses aaa new-model with IOS RADIUS client configuration. The ASA uses the aaa-server command family with separate protocol and host stanzas.

A Linksys WRT300N wireless router provides guest WiFi access. Its WAN port connects to HQ-Dist2 FastEthernet0/7, configured as an access port in VLAN 70. Section 11

Monitoring — SNMP & Syslog

11.1 SNMP — Simple Network Management Protocol

SNMPv2c is configured on all routing and switching devices using the community string SecureNet-RO in read-only mode. The SNMP manager is SRV-Internal (10.0.1.10), which can poll device MIBs for interface statistics, CPU utilisation, memory usage, and routing table information. This enables proactive monitoring and capacity planning.

Devices with SNMP configured: HQ-ASA, HQ-Core, HQ-Dist1, HQ-Dist2, BR1-Router, BR2-Router — all using community string SecureNet-RO RO pointing to collector 10.0.1.10.

11.2 Syslog

All network devices forward syslog messages to SRV-Internal (10.0.1.10). Syslog provides a centralised audit trail of device events including interface state changes, routing adjacency changes, ACL hits, NAT translations, and authentication attempts. Combined with NTP synchronisation, log timestamps are reliable across all devices, enabling accurate incident timeline reconstruction.

Section 12

SDN & Cloud Trends (Part D)

12.1 Software-Defined Networking

Software-Defined Networking (SDN) decouples the network control plane from the data plane. In traditional networks like the one implemented in this project, each device independently makes forwarding decisions based on locally held routing tables. In an SDN architecture, a centralised controller (such as Cisco DNA Center or an OpenDaylight controller) maintains a global view of the network and pushes forwarding rules to individual devices.

For SecureNet Technologies, an SDN migration would offer several benefits: centralised policy enforcement (eliminating the need to configure individual devices), dynamic traffic engineering, faster network programmability via APIs (northbound REST interfaces), and simplified network visualisation. Technologies such as VXLAN with EVPN control plane would replace the traditional VLAN/STP model, removing the broadcast domain limitations inherent in the current design.

12.2 Cloud Networking Trends

Cloud adoption is transforming enterprise network design. Rather than hosting all services in an on-premises data centre (as implemented here with SRV-Internal, DNS-Server, AAA-Server), organisations increasingly leverage cloud service providers for compute and storage while maintaining on-premises connectivity via SD-WAN or direct cloud connects.

Key trends relevant to SecureNet's future evolution include:

  • SD-WAN: Replacing the IPsec VPN model with an SD-WAN overlay that provides application-aware routing, automatic failover between WAN connections, and centralised policy management — removing the need for manual crypto map configuration on each branch router
  • SASE (Secure Access Service Edge): Converging network security (firewall, CASB, ZTNA) with WAN connectivity into a cloud-delivered service, eliminating the need for on-premises ASA hardware as traffic inspection moves to the cloud edge
  • Zero Trust Architecture: Moving away from perimeter-based security (the current model where internal traffic is implicitly trusted) toward identity-based micro-segmentation where every connection is authenticated and authorised regardless of network location
  • Infrastructure as Code: Managing network configurations via tools like Ansible, Terraform, or Cisco NSO rather than manual CLI, enabling version-controlled, repeatable, and auditable network changes

12.3 Governance Considerations

Network design at SecureNet must align with applicable governance frameworks. The VLAN segmentation and firewall ACLs implemented in this design directly support compliance with principles from ISO 27001 (information security management) and frameworks such as NIST SP 800-53. Specific controls addressed include network segmentation (SC-7), identification and authentication (IA-2 via AAA/RADIUS), audit logging (AU-2 via Syslog), and time synchronisation (AU-8 via NTP). Regular penetration testing of the ASA ACLs and VPN configuration would be recommended as part of ongoing governance.

Section 13

Conclusion

This project has successfully designed and simulated a comprehensive enterprise network for SecureNet Technologies that meets all stated business and technical requirements. The implemented solution delivers:

  • High availability through dual distribution switches with HSRP gateway redundancy, EtherChannel peer-link, dual-uplink access switches, and STP failover
  • Security depth through a three-zone ASA firewall, ACL-enforced DMZ, encrypted VPN tunnels, AAA-controlled device access, and guest network isolation
  • Scalable routing through hierarchical OSPF at HQ, RIPv2 at branches, and a documented BGP upgrade path for internet edge connectivity
  • Operational visibility through centralised SNMP monitoring, Syslog aggregation, NTP synchronisation, and DNS name resolution
  • Service richness through dedicated servers for DNS, AAA, NTP/Syslog/SNMP, corporate wireless, and guest WiFi isolation

The network has been fully validated in Cisco Packet Tracer 9.0 with 45 devices and 52 physical links, zero blocking validation errors, and complete configuration on all devices. The design provides a solid foundation for SecureNet Technologies' current operations while establishing clear upgrade paths toward SDN, SD-WAN, and cloud-native architectures as the organisation grows.
